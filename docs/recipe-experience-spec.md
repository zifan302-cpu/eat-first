# 菜谱体验规格

状态：阶段 A 已实现并通过本地验证，阶段 B、C 尚未开发
适用范围：V0.7 之后的菜谱体验迭代
产品边界：单人、本地优先；菜谱服务于食材处理闭环，不扩张为完整菜谱社区或家庭 ERP

## 1. 目标

菜谱功能的任务不是单纯回答“这些食材能做什么”，而是帮助用户在现实条件下完成一次食材处理决定：

1. 系统先判断今天值得优先处理什么。
2. 用户用很少的操作补充本次条件。
3. 系统提供两个取舍明确、可以实际完成的方案。
4. 用户可以局部调整，而不必重新填写全部条件。
5. 做完后确认实际用量，并更新冰箱中的剩余数量。

默认体验应在不修改任何选项时完成生成；复杂控制只在用户主动展开后出现。

## 2. 非目标

- 不建设公开菜谱社区、关注、评论或排行榜。
- 不把营养、医疗或食品安全判断包装成可靠结论。
- 不默认加入购物清单、外卖或生鲜购买链路。
- 不要求用户在每次生成前管理完整库存或逐项选择所有食材。
- 不使用开放式聊天作为唯一调整方式。
- 不在用户确认前自动扣减食材数量。
- 不因为用户没有处理某项食材而施加惩罚。

## 3. 体验原则

### 3.1 系统先判断，用户负责纠正

系统根据日期、开启状态和现有优先级逻辑给出建议。用户可以移除、替换或明确指定，但不需要从空白状态开始配置。

### 3.2 建议优先不等于强制混合

临期食材是推荐目标，不是必须全部塞进一道菜。只有用户主动标记“必须使用”时，模型才承担严格使用约束。

### 3.3 一次只暴露一层复杂度

长期偏好、本次条件和系统判断必须分层。默认界面只显示本次最可能变化的内容，其余内容通过摘要和“调整”入口呈现。

### 3.4 两个方案必须体现真实取舍

两个方案不能只是同一道菜的轻微变体。它们应在速度、食材覆盖、洗锅数量、是否适合便当或是否批量制作等方面形成可解释差异。

### 3.5 菜谱不是终点，处理食材才是终点

最终成功动作是“确认做了什么并更新冰箱”，而不是生成一段文本。

## 4. 三层信息架构

### 4.1 长期偏好

保存在本地设置中，生成时自动带入：

- 界面与菜谱输出语言
- 默认菜系倾向
- 默认人数和可接受时间
- 长期饮食限制与过敏原
- 已拥有的厨具
- 常备调料与基础食材策略

### 4.2 本次条件

只影响当前请求，不自动改写长期偏好：

- 本次人数和时间
- 本次做饭目标
- 本次饮食要求
- 本次不想使用的食材
- 本次必须使用的食材
- 本次不想启用或只想启用的厨具

### 4.3 系统判断

默认不要求用户配置：

- 今日建议优先食材
- 可用于搭配的食材范围
- 两个方案的差异方向
- 食材与时间是否现实匹配
- 哪种厨具组合更合适

## 5. 推荐主流程

```text
首页菜谱入口
  → 轻量生成面板
  → 生成中
  → 两个差异化方案
  → 选择或局部调整
  → 做饭步骤
  → 确认实际用量
  → 更新冰箱并记录结果
```

生成失败、超时或取消时，面板中的本次条件必须保留。

## 6. 生成前界面

### 6.1 默认状态

默认面板只显示：

```text
今天建议先用
番茄 · 蘑菇 · 蓝莓                         调整

冰箱还有 8 件食材可供搭配
AI 会自行选择合适组合

本次需求（可选）
[例如：少辣、不要奶制品、想吃清淡一点]

1 人 · 30 分钟 · 中式日常                 修改
按炉灶、电饭煲生成                        临时调整

[生成两个方案]
```

默认不展示完整食材列表、厨具复选框或多个大型下拉框。

### 6.2 本次做饭目标

“修改”区域可以提供单选目标：

- 自动判断
- 尽快吃上
- 多处理临期食材
- 少洗锅
- 做成便当
- 一次多做一些
- 不开火

目标一次最多选择一个，避免形成互相冲突的要求。饮食限制和过敏信息优先级始终高于做饭目标。

### 6.3 食材调整面板

用户可以浏览和搜索全部符合菜谱条件的当前食材。每项食材只有三种本次角色：

- 建议优先：系统推荐，模型尽量使用。
- 必须使用：用户明确要求，模型必须在至少一个方案中使用。
- 这次不用：不发送给模型或明确列入排除项。

其余食材默认为“可以使用”，无需逐项选中。

界面不得只展示固定五项而隐藏其他食材。默认摘要可以折叠，但高级面板需要覆盖完整可用冰箱列表。

## 7. 食材选择与请求边界

### 7.1 食材资格

只有状态为 active 的食材可以进入菜谱上下文。明确过期的 Use by 食材继续排除，不允许通过用户操作重新加入菜谱。

需要检查品质的 Best before 或已开启食材可以进入，但结果中只显示一次简短、条件式提醒，不重复插入安全说教。

### 7.2 自动排序

候选食材按以下顺序在本地排序：

1. 用户明确标记必须使用。
2. 今日需要处理。
3. 即将到期或开启后临近处理日期。
4. 需要检查品质。
5. 普通可用食材。
6. 无日期食材。

同一等级继续沿用当前优先级分数和日期排序。

### 7.3 数量边界

- 建议优先食材：默认最多 3 项。
- 用户必须使用：建议最多 3 项，超出时提示要求可能互相冲突。
- 自动发送的可以使用食材：默认最多 15 项。
- 用户主动指定的食材不应因自动上限被截断。
- 超过 15 项时，界面仍可浏览全部食材；请求端按优先级选择，并保证用户操作优先。

输入成本主要通过精简字段控制，而不是通过把用户锁死在五个选项中。

### 7.4 建议请求结构

```ts
interface RecipeRequestV2 {
  locale: "zh-CN" | "en-GB";
  cuisine: "auto" | "chinese_home" | "global_everyday";
  servings: number;
  maxMinutes: number;
  cookingGoal:
    | "auto"
    | "fast"
    | "rescue_more"
    | "one_pan"
    | "lunchbox"
    | "batch_cook"
    | "no_cook";
  dietaryNotes?: string;
  ownedEquipment: CookingEquipment[];
  customEquipment: string[];
  equipmentMode: "use_any_owned" | "only_selected";
  selectedEquipment?: CookingEquipment[];
  pantryProfile: PantryProfile;
  customPantryStaples: string[];
  suggestedFoods: RecipeFoodInput[];
  requiredFoods: RecipeFoodInput[];
  availableFoods: RecipeFoodInput[];
  excludedFoodIds: string[];
}
```

`requiredFoods` 只能来自用户明确操作，不能由系统自动产生。

## 8. 厨具系统

### 8.1 长期档案与本次使用分离

设置中记录“我拥有的厨具”；菜谱面板默认使用这些信息生成。用户可以临时选择：

- 本次不使用某件厨具。
- 本次只使用选中的厨具。
- 本次临时增加一件设备。

拥有烤箱不代表用户今天愿意预热烤箱，因此必须支持临时覆盖。

### 8.2 推荐厨具分类

基础加热：

- 炉灶或电磁炉
- 烤箱
- 微波炉
- 空气炸锅
- 电烤盘或电饼铛
- 户外烧烤炉

锅具型设备：

- 电饭煲
- 蒸锅
- 电压力锅
- 多功能料理锅
- 慢炖锅
- 砂锅或炖锅

加工设备：

- 台式搅拌机
- 手持搅拌棒
- 食物处理机
- 破壁机

轻食设备：

- 烤面包机
- 三明治机
- 华夫饼机

允许使用“添加其他厨具”兼容地区性设备。自定义厨具最多 8 项，每项最多 24 个字符，需要去重并作为不可信输入处理。

### 8.3 默认规则

- 新用户默认只假设炉灶和基础锅具。
- 模型不得使用用户未拥有的特殊厨具。
- 菜谱结果必须列出实际需要的厨具。
- “不开火”目标不得使用炉灶、烤箱、空气炸锅或压力设备。

## 9. 常备调料与基础食材

设置页提供简化档案，而不是建立另一套完整库存：

- 油、盐、糖
- 酱油、醋
- 黑胡椒和常用干香料
- 面粉、淀粉
- 黄油或常用烹饪脂肪
- 常用米、面或面包

建议提供三种策略：

- 严格：只使用明确标记拥有的常备项。
- 日常：允许建议少量普通常备项，并清楚列出。
- 灵活：允许额外建议，并提供替代方案。

新用户默认使用“日常”。葱姜蒜、牛奶、鸡蛋等可能过期的食材仍应作为普通冰箱食材管理，不默认视为无限常备。

固定列表之外可以保存最多 12 个自定义常备项，每项最多 24 个字符。自定义常备项只记录名称，不记录数量、日期或包装；会变化或会过期的食材仍必须进入冰箱库存。

## 10. 生成中状态

- 展示预计等待状态，但不伪造百分比进度。
- 允许取消。
- 取消后保留全部本次条件。
- 超时后提供“减少限制后重试”和“保持条件重试”。
- 同一请求短时间内优先复用本地缓存，避免重复调用。
- 用户离开弹窗时中止前端请求，不让旧结果覆盖新请求。

## 11. 结果比较页

每个方案首先展示决策信息，而不是直接堆叠长步骤：

- 菜名
- 核心差异标签
- 真实总用时
- 使用的冰箱食材
- 预计使用数量
- 需要但冰箱中没有的普通食材
- 需要的厨具
- 一句话说明为什么推荐

推荐差异标签：

- 最快完成
- 多处理食材
- 少洗锅
- 适合便当
- 适合多做一份
- 无需开火

标签只能根据实际方案内容产生，不允许作为随机装饰。

两个方案至少在主要做法、时间取舍或使用食材组合中有一个明显差异。若模型返回两个近似方案，应在服务端修复或只重生成第二个方案。

## 12. 局部调整

每个方案提供受控调整入口：

- 不要这个食材
- 没有某种调料
- 换一种主要做法
- 缩短用时
- 改成便当
- 减少洗锅
- 调整人数
- 只重新生成这个方案

局部调整应携带原方案和新增约束，只替换被调整的方案。另一个方案保持不变，减少等待与调用量。

开放式补充文本可以作为二级入口，但不能替代上述明确操作。

## 13. 做饭模式

选定方案后进入单方案页面：

- 顶部固定显示菜名、总时间和人数。
- 食材清单可逐项勾选。
- 步骤使用 4—6 个清晰阶段。
- 时间步骤可以启动本地计时器。
- 支持保持屏幕唤醒属于后续增强，不作为首版必需。
- 随时可以返回方案比较页，不丢失结果。

## 14. 做完后的冰箱更新

点击“我做了这个”后，不立即修改库存，而是展示用量确认：

```text
这次实际用了：

番茄       2 / 4 个
蘑菇       150 / 250 克
菠菜       全部用完

[确认并更新冰箱]
```

规则：

- 有结构化数量时，允许用户修改实际用量。
- 只有自由文本数量时，提供“全部用完”“用了一些”“没有使用”。
- 全部用完映射为 eaten。
- 部分使用沿用现有 partially_used 行为。
- 没有使用不修改食材。
- 整次更新形成一个可撤销操作。
- 模型估算数量只能作为默认值，不能直接成为库存事实。

## 15. 不想做饭的替代出口

在生成入口和结果页保留弱化入口“今天不想做饭，先处理食材”。可建议：

- 冷冻
- 分装
- 焯水或简单预处理
- 制作基础酱料
- 按包装说明调整计划
- 明日优先提醒

该能力服务于减少浪费，不强迫所有临期食材都进入一道菜。

## 16. 历史、收藏与本地学习

### 16.1 菜谱历史

本地保存最近生成和实际完成的方案，包含生成条件、选择结果和是否完成。历史记录不得自动恢复已经被删除的食材。

### 16.2 快速反馈

- 喜欢这个做法
- 不喜欢这种做法
- 太复杂
- 时间估计不准
- 以后少推荐这种菜系

反馈保存在本地，用于调整后续排序和提示词，不建设公开评分体系。

### 16.3 收藏

收藏的是做法模板，不是绑定某一批即将过期食材的静态快照。再次使用时需要重新匹配当前冰箱。

## 17. 模型契约

提示词必须同时约束：

1. 输出语言由 locale 决定，菜系由 cuisine 决定，两者互不覆盖。
2. requiredFoods 必须覆盖；suggestedFoods 尽量跨两个方案覆盖。
3. availableFoods 只在改善方案时使用，不必强行加入。
4. excludedFoodIds 绝不能重新出现。
5. 不使用未允许的特殊厨具。
6. 至少一个方案满足 maxMinutes。
7. 两个方案需要明显差异。
8. 饮食限制优先于做饭目标和食材利用率。
9. 内部 ID、日期字段和优先级标签不得进入用户可见文本。
10. 不声称食材安全，也不重新引入明确过期的 Use by 食材。

建议结果结构增加：

```ts
interface RecipeOptionV2 {
  id: string;
  title: string;
  summary: string;
  differenceTags: RecipeDifferenceTag[];
  whyThisOption: string;
  totalMinutes: number;
  servings: number;
  ingredients: string[];
  steps: string[];
  equipment: CookingEquipment[];
  usesFoods: Array<{
    foodId: string;
    estimatedAmount?: number;
    estimatedUnit?: FoodQuantityUnit;
  }>;
  missingIngredients: string[];
}
```

## 18. 结果校验与修复

服务端展示前检查：

- 是否恰好有两个方案
- 输出语言是否正确
- requiredFoods 是否被覆盖
- excludedFoods 是否泄漏
- 是否使用未允许厨具
- 是否至少一个方案满足时间要求
- 两个方案标题和主要步骤是否过度相似
- 内部 ID 是否泄漏
- 食材和步骤字段是否完整

单个方案不合格时优先只修复该方案。结构损坏时可以进行一次低温度修复；仍失败则返回可理解错误，不展示残缺菜谱。

## 19. 性能与调用策略

- 默认使用快速模型完成生成。
- 只有结构修复、复杂限制冲突或连续失败时才考虑更强模型。
- 对规范化后的相同请求生成短期缓存键。
- 局部调整只发送必要的原方案与变化条件。
- 输出长度保持两个方案、每个 4—6 步，不生成冗长文章。
- 请求取消、超时和旧响应覆盖必须有自动测试。

## 20. 页面状态

菜谱体验至少覆盖以下状态：

- 无可用食材
- 只有一项可用食材
- 建议优先食材与普通食材并存
- 冰箱食材超过 20 项
- 所有候选均被用户排除
- 用户指定多个必须使用食材
- 没有配置特殊厨具
- 本次只允许微波炉或不开火
- 有长期饮食限制和本次临时要求
- 生成中、取消、超时、限流、服务未配置
- 两个方案正常返回
- 单个方案修复
- 做完后全部用完、部分使用和没有使用

## 21. 验收标准

### 默认效率

- 已有食材和默认偏好时，用户无需展开高级设置即可发起生成。
- 从打开面板到发起生成最多需要一次主要按钮点击。
- 默认面板在 430 × 900 视口中能看到食材摘要、本次需求和生成按钮。

### 灵活性

- 用户可以在高级面板找到全部符合资格的冰箱食材。
- 用户可以区分建议、必须和排除，而不是被固定五个候选限制。
- 厨具档案支持多种设备，同时默认面板不展示完整复选列表。

### 结果质量

- 两个方案存在可观察的差异。
- 至少一个方案满足时间约束。
- 结果明确列出使用食材、缺少食材和所需厨具。
- 中文界面返回中文，英文界面返回英文；菜系设置独立生效。

### 闭环

- 生成、局部调整和取消不会修改库存。
- 只有“确认并更新冰箱”会写入食材状态或数量。
- 更新支持部分使用和撤销。

## 22. 分阶段实施建议

### 阶段 A：减少生成前负担

- 轻量默认面板与渐进展开
- 建议、必须、可用、排除四类食材语义
- 自动候选从 5 项提高到最多 15 项
- 多样化厨具档案和本次临时覆盖
- 常备调料策略
- 新请求结构、提示词和结果校验

### 阶段 B：提高结果可选择性

- 方案差异标签
- 所需厨具、缺少食材和预计使用量
- 单方案重生成与受控调整
- 请求缓存和菜谱历史

### 阶段 C：完成冰箱闭环

- 做饭模式
- 实际用量确认
- 批量更新、部分使用和整次撤销
- 收藏与本地偏好反馈
- 不做饭时的冷冻和预处理建议

阶段 A 完成并验证后再决定是否同时进入阶段 B。阶段 C 不应在食材数量模型和部分使用行为稳定前开发。

## 23. 当前 V0.7 与目标差异

| 当前 V0.7 | 下一阶段目标 |
|---|---|
| 三项 priorityFoods 作为强约束 | suggestedFoods 为软建议，requiredFoods 只来自用户明确操作 |
| 最多五项 availableFoods | 界面可访问全部，自动请求最多 15 项 |
| 每次生成展示完整厨具选择 | 设置保存拥有的厨具，生成页只显示摘要与临时调整 |
| 饮食要求作为单次文本 | 长期限制与本次要求分离，单次输入仍保留 |
| 两个基础菜谱结果 | 两个具有差异标签和现实取舍的可比较方案 |
| 生成后与库存脱节 | 用户确认实际用量后更新冰箱并支持撤销 |

本文件确定体验方向，不代表上述目标已经上线。
